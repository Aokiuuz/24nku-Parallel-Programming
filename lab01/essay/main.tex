\section{实验概述}
\subsection{实验目的}
本实验对应 Lab01「体系结构相关编程」，围绕 CPU cache 局部性、指令级并行和编译器优化三个主题，完成两类微基准程序的实现、验证与性能测试：
\begin{enumerate}
  \item 矩阵与向量内积：比较逐列访问的平凡算法与按行访问的 cache 优化算法，并进一步尝试循环展开。
  \item $n$ 个数求和：比较单链式累加与多路链式累加，观察减少数据相关后对超标量执行效率的影响。
  \item 对比不同编译优化等级（\texttt{-O0}、\texttt{-O2}、\texttt{-O3}）下的运行结果，并结合 VTune 与 Godbolt 对性能来源进行剖析。
\end{enumerate}

\subsection{研究问题与待验证假设}
结合 Lab1 文档中的题目分析，本报告重点回答以下三个问题：
\begin{enumerate}
  \item 对矩阵题而言，改变访存顺序是否能显著改善 cache 行为，并且这种优势是否会随着工作集增大而继续扩大？
  \item 对求和题而言，多累加器与循环展开是否能有效降低累加依赖链长度，从而在 \texttt{-O2/-O3} 下提高超标量利用率？
  \item 编译器优化、算法设计和 profiling 结果之间是否能形成一致的证据链，解释实验中的主要性能差异？
\end{enumerate}

对应地，本文待验证的假设为：矩阵题的主要瓶颈来源于访存模式不匹配，求和题的主要瓶颈来源于串行依赖链，而编译器高优化等级会进一步放大这两类手工优化的收益。

\subsection{实验完成情况}
结合 Lab01 要求，当前已经完成的内容如下：
\begin{itemize}
  \item 基础要求：矩阵题完成 \texttt{col\_naive}、\texttt{row\_cache}；求和题完成 \texttt{sum\_naive}、\texttt{sum\_dual}。
  \item 进阶要求：完成循环展开版本 \texttt{row\_cache\_u4} 与 \texttt{sum\_unroll4}。
  \item 补充算法：实现树形求和 \texttt{sum\_pairwise}，用于对比更多算法设计思路。
  \item 编译器优化对比：已经在 \texttt{-O0}、\texttt{-O2}、\texttt{-O3} 三种编译优化等级下完成基准测试。
  \item profiling 分析：已经完成矩阵题的 VTune Hotspots / Memory Access 分析、求和题的 VTune Hotspots / Microarchitecture 分析，以及两组算法的 Godbolt 汇编对比分析。
  \item 实验源码 GitHub 链接：\url{https://github.com/Aokiuuz/24nk-Parallel-Program-Design.git}
\end{itemize}

\section{实验设计与实现}
\subsection{矩阵与向量内积}
设矩阵 $B \in \mathbb{R}^{n \times n}$、向量 $a \in \mathbb{R}^{n}$，计算目标为：
\begin{equation}
  \text{sum}[i] = \sum_{j=0}^{n-1} B[j][i] \cdot a[j], \quad 0 \le i < n
\end{equation}
由于 C/C++ 中二维数组按行主序存储，若按列访问矩阵，会造成跨 cache line 的非连续读；若按行访问，则更符合空间局部性。

\subsubsection{算法设计}
\begin{enumerate}
  \item \texttt{col\_naive}：固定列索引 $i$，内层枚举行索引 $j$，直接按数学定义实现。
  \item \texttt{row\_cache}：固定行索引 $j$，把当前 $a[j]$ 缓存在寄存器中，并顺序扫描该行全部元素，降低跨行跳跃访问带来的 cache miss。
  \item \texttt{row\_cache\_u4}：在 \texttt{row\_cache} 基础上对内层循环做四路展开，减少循环控制与索引更新开销。
\end{enumerate}

\begin{breakablealgorithm}
  \caption{矩阵列内积的 cache 优化思路}
  \begin{algorithmic}[1]
    \Require 行主序存储矩阵 $B$，向量 $a$，输出数组 $\text{sum}$，规模 $n$
    \Ensure 输出每一列与向量 $a$ 的内积
    \State 将 $\text{sum}[0 \ldots n-1]$ 置零
    \For{$j = 0$ to $n-1$}
      \State $aj \gets a[j]$
      \State $base \gets j \times n$
      \For{$i = 0$ to $n-1$}
        \State $\text{sum}[i] \gets \text{sum}[i] + B[base + i] \times aj$
      \EndFor
    \EndFor
    \State \Return $\text{sum}$
  \end{algorithmic}
\end{breakablealgorithm}

\subsubsection{关键实现}
\begin{lstlisting}[title=矩阵题中按行访问的核心实现,frame=trbl,language={C++}]
void row_cache(const Matrix& b, const Vector& a, Vector& sum, std::size_t n) {
    std::fill(sum.begin(), sum.end(), 0.0);
    for (std::size_t j = 0; j < n; ++j) {
        const double aj = a[j];
        const std::size_t base = j * n;
        for (std::size_t i = 0; i < n; ++i) {
            sum[i] += b[base + i] * aj;
        }
    }
}
\end{lstlisting}

\subsubsection{规模设置说明}
矩阵题选择 $n=128,256,512,1024$ 四个规模，考虑如下：
\begin{enumerate}
  \item 这四组规模覆盖了从较小工作集到较大工作集的增长过程，足以观察 cache 局部性对性能的影响是否随规模扩大而增强；
  \item 随着 $n$ 增大，矩阵数据量按平方增长，更容易放大“按列访问”和“按行访问”的访存差异；
  \item 重复次数采用分段递减策略，目的是在保证总耗时可接受的同时，让不同规模下的测量仍然具有稳定性。
\end{enumerate}

\subsection{\texorpdfstring{$n$}{n} 个数求和}
求和实验关注的不是算法复杂度，而是累加过程中的数据相关。单链式累加存在严格的串行依赖，而多路累加可把长依赖链拆成多条较短的依赖链，更有利于处理器发掘指令级并行。

\subsubsection{算法设计}
\begin{enumerate}
  \item \texttt{sum\_naive}：使用单个累加器顺序累加，作为基准实现。
  \item \texttt{sum\_dual}：使用两个累加器分别处理偶数位与奇数位，减少单一路径的相关长度。
  \item \texttt{sum\_unroll4}：采用四个累加器并配合四路循环展开。
  \item \texttt{sum\_pairwise}：使用树形规约思路，两两相加后写回数组，体现不同设计思路，但会引入额外访存。
\end{enumerate}

\begin{breakablealgorithm}
  \caption{四路展开求和}
  \begin{algorithmic}[1]
    \Require 输入数组 $a[0 \ldots n-1]$
    \Ensure 数组所有元素之和
    \State $s_0, s_1, s_2, s_3 \gets 0$
    \State $i \gets 0$
    \While{$i + 3 < n$}
      \State $s_0 \gets s_0 + a[i]$
      \State $s_1 \gets s_1 + a[i + 1]$
      \State $s_2 \gets s_2 + a[i + 2]$
      \State $s_3 \gets s_3 + a[i + 3]$
      \State $i \gets i + 4$
    \EndWhile
    \While{$i < n$}
      \State $s_0 \gets s_0 + a[i]$
      \State $i \gets i + 1$
    \EndWhile
    \State \Return $(s_0 + s_1) + (s_2 + s_3)$
  \end{algorithmic}
\end{breakablealgorithm}

\begin{lstlisting}[title=求和实验中的四路展开实现,frame=trbl,language={C++}]
double sum_unroll4(const Vector& a) {
    double sum0 = 0.0, sum1 = 0.0, sum2 = 0.0, sum3 = 0.0;
    std::size_t i = 0;
    for (; i + 3 < a.size(); i += 4) {
        sum0 += a[i];
        sum1 += a[i + 1];
        sum2 += a[i + 2];
        sum3 += a[i + 3];
    }
    for (; i < a.size(); ++i) {
        sum0 += a[i];
    }
    return (sum0 + sum1) + (sum2 + sum3);
}
\end{lstlisting}

\subsubsection{规模设置说明}
求和题选择 $2^{10}$、$2^{14}$、$2^{18}$、$2^{22}$ 四组输入规模，主要是为了让数据规模跨越多个数量级：
\begin{enumerate}
  \item 小规模时，处理器更可能受益于减少依赖链长度带来的指令级并行提升；
  \item 大规模时，数据访问的带宽与 cache 行为会逐渐成为主导因素，可以观察展开收益是否衰减；
  \item 以 $2^k$ 形式设置规模，也便于把实验现象与 cache 容量、对齐和规约层数联系起来分析。
\end{enumerate}

\begin{enumerate}
  \item 在 \texttt{-O0} 下，\texttt{sum\_dual} 和 \texttt{sum\_unroll4} 普遍慢于 \texttt{sum\_naive}，说明缺少寄存器分配和向量化支持时，多累加器结构本身会带来额外控制开销。
  \item 当优化等级提升到 \texttt{-O2} 和 \texttt{-O3} 后，多路累加优势迅速显现。图\ref{fig:sum-speedup} 和图\ref{fig:sum-best-opt} 显示，在 $2^{10}$ 至 $2^{18}$ 的规模区间内，\texttt{sum\_unroll4} 可稳定获得约 4 倍加速。
  \item \texttt{sum\_unroll4} 在中等规模上表现最佳。例如 \texttt{-O3}、$n=262144$ 时加速比达到 4.339，说明此时“缩短依赖链 + 减少循环控制”的收益仍明显高于额外访存成本。
  \item 当规模达到 $2^{22}$（32 MiB）时，\texttt{sum\_dual} 与 \texttt{sum\_unroll4} 的优势缩小到约 1.35 倍，表明瓶颈已逐渐从依赖链转向内存访问与数据搬运。
  \item \texttt{sum\_pairwise} 在大规模下显著落后，主要原因是树形规约需要反复写回中间结果，带来额外复制和访存开销。
  \item 图\ref{fig:sum-opt} 说明编译器优化本身就能显著加速基线算法，因此评价手工优化时，必须把编译器选项与算法结构结合起来看。
\end{enumerate}

\subsection{综合结论}
结合两组实验，可以得到以下结论：
\begin{enumerate}
 \item 对矩阵题而言，访存顺序决定了 cache 行为。与数据布局一致的顺序访问能够显著降低访存延迟，其收益会随问题规模扩大而增强。
  \item 对求和题而言，算法复杂度虽然相同，但数据相关不同。通过多路累加缩短依赖链，可以显著提高超标量执行效率；这一收益在中等工作集上最明显，在超大工作集上会被访存成本部分抵消。
  \item 编译器优化等级会强烈影响手工优化的最终表现。高优化等级不仅能降低基线算法的执行时间，也能把 cache 友好写法和多累加器结构进一步转化为向量化与更有效的指令调度。
  \item VTune 与 Godbolt 的剖析结果和性能表、趋势图相互印证，因此本文结论不仅来自现象观察，也获得了 profiling 与汇编层面的支持。
\end{enumerate}

\section{Godbolt 汇编分析}
\subsection{分析方法}
为了进一步解释性能测试中的差异，本文使用 Godbolt（Compiler Explorer）观察关键函数在不同优化选项下生成的汇编代码，主要对比如下几组函数：
\begin{itemize}
  \item 矩阵题：\texttt{col\_naive}、\texttt{row\_cache}、\texttt{row\_cache\_u4}
  \item 求和题：\texttt{sum\_naive}、\texttt{sum\_dual}、\texttt{sum\_unroll4}、\texttt{sum\_pairwise}
\end{itemize}
编译器选用 x86-64 GCC，主要考察 \texttt{-O0}、\texttt{-O3} 和 \texttt{-O3 -march=native} 三组编译选项，分析重点包括：
\begin{enumerate}
  \item 内层循环的地址计算是否简化；
  \item 是否减少了循环控制与分支开销；
  \item 是否通过多个寄存器暴露更多独立计算链；
  \item 编译器是否自动生成了 SSE/AVX 向量指令。
\end{enumerate}

\subsection{矩阵题汇编分析}
\subsubsection{\texttt{col\_naive} 与 \texttt{row\_cache}}
从 \texttt{-O0} 版本可以直接看出，\texttt{col\_naive} 的内层循环需要反复做索引计算和跨步读，循环体较长；\texttt{row\_cache} 则已经更接近“缓存当前标量 + 顺序扫描当前行”的结构。到了 \texttt{-O3} 和 \texttt{-O3 -march=native}，两者都能生成 SSE/AVX 乘加指令，但 \texttt{col\_naive} 的访存本质仍是跨行取列，而 \texttt{row\_cache} 对应的是连续地址上的向量乘加，因此算法层面的局部性优势会被编译器进一步放大。这与表\ref{tab:matrix} 中 \texttt{row\_cache} 和 \texttt{row\_cache\_u4} 明显优于 \texttt{col\_naive} 的结果一致。

\subsubsection{\texttt{row\_cache} 与 \texttt{row\_cache\_u4}}
从汇编上看，\texttt{row\_cache\_u4} 的主要作用是进一步减少循环控制开销。高优化下普通 \texttt{row\_cache} 已经被自动优化，因此两者的汇编差距并不大，这也解释了为什么 \texttt{row\_cache\_u4} 往往更快，但收益远小于“按列改按行”带来的提升。

因此，矩阵题的 Godbolt 结论可以概括为：决定性优化首先来自访问顺序的改变，循环展开属于次一级优化；编译器在 \texttt{-O3} 及以上会放大这种优势，但无法从根本上修复不良的访存模式。

\subsection{求和题汇编分析}
\subsubsection{\texttt{sum\_naive} 与 \texttt{sum\_dual} / \texttt{sum\_unroll4}}
求和题中最关键的现象是“累加依赖链”的变化。\texttt{sum\_naive} 在 \texttt{-O3} 下虽然已经减少了部分循环开销，但核心仍围绕单个累加寄存器形成长依赖链；\texttt{sum\_dual} 和 \texttt{sum\_unroll4} 则在源码层面显式提供了多条独立累加路径。尤其是 \texttt{sum\_unroll4}，在 \texttt{-O3} 及 \texttt{-O3 -march=native} 下可以看到更明显的 SIMD 累加指令和更宽寄存器使用，说明它更容易同时利用超标量调度与向量化能力。这与表\ref{tab:sum} 中 \texttt{sum\_dual}、\texttt{sum\_unroll4} 在 \texttt{-O2/-O3} 下显著优于 \texttt{sum\_naive} 的结果一致。

\subsubsection{\texttt{sum\_unroll4} 与 \texttt{sum\_pairwise}}
\texttt{sum\_pairwise} 的汇编则体现出另一类思路：它不是围绕单次扫描构建多累加器，而是反复对 \texttt{g\_work} 做“读-加-写回”的两两规约。即使在 \texttt{-O3 -march=native} 下，这种结构仍明显依赖内存访问，难以像 \texttt{sum\_unroll4} 那样把主要工作保留在寄存器中完成，因此在大规模下明显更慢。

\subsection{Godbolt 分析结论}
结合以上汇编观察，可以得到如下结论：
\begin{enumerate}
  \item 对矩阵题而言，\texttt{row\_cache} 的核心优势首先来自顺序访存，而不是单纯依赖编译器自动优化。
  \item 对求和题而言，\texttt{sum\_dual} 和 \texttt{sum\_unroll4} 的收益主要来自减少单一累加器上的串行依赖，使编译器更容易生成多路并行累加和 SIMD 指令。
  \item \texttt{sum\_pairwise} 说明算法形式不同并不必然更快；若实现中引入大量中间结果写回，访存开销会抵消其结构优势。
  \item Godbolt 观察到的汇编特征与前文性能表中的规律基本一致，因此可以作为解释实验结果的底层证据。
\end{enumerate}
