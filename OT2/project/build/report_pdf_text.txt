摘要
协程（coroutine）是一种降低等待资源成本的并发抽象，主要用于减少大量 I/O 等待任务对 OS
线程的长期占用。现代云服务中，请求经常在等待网络、数据库、RPC 下游、缓存、消息队列或定
时器；如果每个请求都绑定一个平台线程，那么线程栈、线程对象、上下文切换和调度成本会随着
并发数增加而放大。
本报告围绕四个问题展开：
 1. 并发执行机制为什么会从进程、线程、事件循环演进到协程；
 2. 协程在底层如何实现，包括状态机、协程帧、挂起与恢复；
 3. C++20、Go、Python、C#、Java 21 对协程或轻量并发的支持有何差异；
 4. 本地实验如何验证协程在 I/O 等待场景中的资源优势，以及它在 CPU 密集场景中的边界。
核心结论如下：
 • 协程的主要价值是降低 I/O 等待任务的线程占用；其他任务类型仍需选择匹配其瓶颈的执行
   模型。
 • 操作系统仍然调度真实线程；协程由语言、编译器和运行时在用户态组织。
 • 不同语言的 “协程” 抽象层级不同，工程评估需要结合运行时、调度模型和异步生态。
 • 本地 TCP I/O 实验显示，asyncio 能用 0 个业务新增线程承载 5000 个 TCP 请求；线程池可
   以通过增加 worker 获得更低耗时，但代价是显著增加业务线程和 RSS 增量。
 • CPU-bound 实验显示，协程调度不增加 CPU 算力。CPU 密集任务应使用进程池、原生扩展、
   向量化、SIMD 或 GPU 等并行计算方案。


1. 并发模型为什么演进到协程
1.1 高并发服务的瓶颈常常是等待
在很多服务端程序中，一个请求的大部分时间花在等待外部事件上，CPU 计算通常只占较短片段：
 • 等待客户端网络包到达；
 • 等待数据库、Redis、消息队列或 RPC 下游响应；
 • 等待磁盘、对象存储或文件系统；
 • 等待连接池、限流器、定时器或锁。
如果使用 “一个请求一个线程” 的模型，阻塞等待期间线程本身并没有执行有意义的计算，但线
程栈、调度上下文和内核对象仍然存在。当并发数达到几千甚至几万时，线程资源开销会成为限制
系统扩展性的因素。
早期的 C10K 问题集中讨论了单机同时处理一万个客户端连接的挑战。它推动了事件驱动服务器、
非阻塞 I/O、epoll/kqueue/IOCP、异步框架和协程运行时的发展。现代协程技术可以看作是这一
演进链路上的重要抽象：它试图同时保留事件驱动的资源效率和同步代码的可读性。

这一演进体现了不同抽象对不同瓶颈的适配关系。工程选型应先识别任务瓶颈，再选择执行模型。

   • 强隔离任务仍然适合进程。
   • 传统同步服务端代码迁移可考虑 Java 虚拟线程这类轻量线程化抽象。


2. 协程的底层机制
2.1 协程是可暂停、可恢复的函数
普通函数调用后会沿调用栈连续执行到返回；协程函数可以在挂起点保存当前状态，然后把执行权
交还给调度器，之后再从挂起点继续执行。
一个协程系统通常包含：
   • 协程函数：例如 async def、async、co_await、go；
   • 协程对象或任务对象：代表一次尚未完成的异步执行；
   • 挂起点：例如 await、yield、channel 阻塞、sleep、I/O 等待；
   • 调度器：决定哪个可运行协程继续执行；
   • 事件源：网络 I/O、定时器、文件 I/O、锁、Future 完成等。
通俗地说，协程把一个连续执行的函数拆成多个片段。每当执行到 await 这类挂起点，如果等待对
象尚未完成，当前函数的局部变量、下一条指令位置、返回值、异常信息等会被保存到协程帧或状
态机对象中，运行时可以去执行其他任务。

Stackless 的优点是轻量、可由编译器明确转换；缺点是调用链通常需要显式传播 async/await。
Stackful 更接近线程心智，侵入性较低，但栈管理、调度和运行时复杂度更高。


2.3 编译器如何把协程转换为状态机
以 async/await 类语言为例，转换过程通常包括：
  1. 找到协程函数中的挂起点；
  2. 把局部变量、下一步执行位置、返回值和异常保存到协程帧；
  3. 遇到 await 且等待对象尚未完成时，注册恢复逻辑并返回给调度器；
  4. 等待对象完成后，调度器恢复协程，从保存的位置继续执行。
因此，协程并不神秘。它本质上是 “语法糖 + 状态机 + 运行时调度” 的组合。


2.4 操作系统仍然调度线程
一个常见误解是 “协程绕过了操作系统”。准确地说：
  • OS 调度的是内核线程或进程，不直接理解 Python Task、Go goroutine、Java virtual thread
    等高层任务；
  • 协程运行时在用户态保存和恢复任务；
  • 协程等待 I/O 时，运行时把 I/O 注册给 OS 的事件通知机制；
  • I/O 完成后，OS 通知运行时，运行时再把对应协程放回就绪队列。
典型 OS I/O 事件机制包括 Linux epoll、BSD/macOS kqueue、Windows I/O Completion Ports，
以及跨平台事件循环库 libuv。


3. 主流语言支持对比
3.1 C++20 coroutine
C++20 引入 co_await、co_yield、co_return 等协程语法。它提供底层协程机制，完整运行时和
调度策略通常由库组织。
  • 编译器负责生成协程帧和恢复逻辑；
  • promise_type、awaiter、coroutine handle 等机制允许库作者高度定制；
  • 标准语言层面没有统一事件循环；
  • 工程使用通常依赖第三方库或平台库，例如 asio、cppcoro、folly 等。
这一路线适合高性能系统库、网络框架、游戏服务器等，但学习成本和生命周期管理风险较高。


3.2 Go goroutine
Go 使用 go f() 启动 goroutine。goroutine 是由 Go runtime 调度的轻量执行单元，其语义与传
统线程和简单协程均有差异，通常配合 channel、context、select 使用。
关键点：
  • goroutine 初始栈小，并可按需增长；
  • Go runtime 使用 M/P/G 模型组织调度；
  • channel 提供通信和同步；
  • context 常用于取消、超时和请求级传播。
Go 的优点是并发语法轻量，运行时支持成熟；goroutine 仍会消耗内存和调度资源，无界创建和
goroutine 泄漏会导致内存、调度和下游压力。



                                    3
3.3 Python asyncio
Python 的 asyncio 通过 event loop、coroutine、Task、Future 组织异步并发。
  • async def 定义协程函数；
  • await 是明确挂起点；
  • asyncio.create_task() 或 TaskGroup 将 coroutine 排入事件循环；
  • 适合高并发 I/O，如 HTTP 客户端、WebSocket、爬虫和异步 Web 服务。
需要注意的是，asyncio 是协作式调度。CPU 长循环或阻塞调用会卡住事件循环。CPU 密集任务通
常需要进程池、原生扩展或外部计算系统。


3.4 C# async/await
C# 的 async/await 与.NET 的 Task 模型结合紧密。编译器会把 async 方法转换成状态机；await
未完成任务时让出执行权，任务完成后继续执行。
这种模型适合应用层 I/O 异步。需要避免在异步代码中随意 .Result 或 .Wait()，否则可能造成线
程阻塞甚至死锁。


3.5 Java 21 Virtual Threads
Java 21 通过 JEP 444 正式引入虚拟线程。该路线保留 “一个任务一个线程” 的同步代码风格，并
在高并发 I/O 场景下提升扩展能力。
  • 虚拟线程由 JVM 调度到平台线程执行；
  • 当虚拟线程执行可卸载的阻塞 I/O 时，JVM 可以把它从承载线程上卸载；
  • 对传统 thread-per-request 服务端代码，迁移成本低于改写成复杂的 Future 链。
虚拟线程主要提升 I/O 并发承载能力。数据库连接池、下游容量、限流、pinning 和 native 调用等
边界仍然需要工程治理。


4. 实验设计与环境
4.1 为什么修正原实验
原始实验使用 asyncio.sleep() 对比 time.sleep()，能够说明等待任务的线程复用机制，但没有
覆盖 socket 连接、发送和读取响应等真实 I/O 路径。因此，该实验保留为概念演示，主证据改用
本地 TCP I/O。
本次修正后，主实验改为本地 TCP I/O：
  • 在本机启动 asyncio TCP server；
  • server 监听 127.0.0.1 动态端口；
  • 每次收到 PING 请求后等待 0.02 秒，再返回 OK；
  • asyncio client 使用 asyncio.open_connection；
  • thread client 使用阻塞 socket 和 ThreadPoolExecutor；
  • 每组重复 5 次，主读中位数。
为了避免 Windows 本地大量短连接触发临时端口和 TIME_WAIT 限制，最终实验采用 “每个并发
执行单元持久连接、多次请求” 的方式。每次请求均经过真实 TCP socket 读写和服务端等待，并
通过持久连接复用降低重复建连影响。

这组结果需要结合耗时与资源成本共同解读。在本实验参数下，线程池增加 worker 后可以显著降
低耗时，尤其是 threads-200；相应代价是新增约 200 个业务线程，并带来更高 RSS 增量。协程
的优势体现在低线程成本下承载同等请求量。

6. 工程实践与常见问题
6.1 适合使用协程的场景
 • 高并发 Web 服务；
 • RPC、API 网关、下游聚合；
 • 爬虫和批量 HTTP 请求；
 • 数据库、缓存、消息队列访问；
 • WebSocket、IM、推送等长连接服务；
 • 大量定时器和超时控制。




                            8
6.2 协程收益有限的场景
 • 纯 CPU 计算；
 • 依赖大量阻塞式第三方库且无法替换；
 • 强实时调度任务；
 • 需要进程级隔离或崩溃隔离的任务；
 • 无界创建任务且没有限流、背压和取消机制的系统。


6.3 工程实践清单
推荐做法：
 • 给外部调用设置超时；
 • 使用结构化并发管理任务生命周期；
 • 对并发数设置上限，例如 semaphore、连接池、令牌桶；
 • 明确取消传播和资源清理；
 • CPU 密集逻辑移出事件循环；
 • 建立监控：任务数、队列长度、事件循环延迟、超时率、取消率、下游错误率。
避免：
 • 无限创建 coroutine、Task 或 goroutine；
 • 在事件循环中直接调用阻塞函数；
 • 忽略取消后的资源清理；
 • 只测平均耗时，不看尾延迟和资源成本；
 • 误把协程当成 “比线程更快的计算单元”。


7. 局限性
本报告实验有以下局限：
 • 本地 loopback TCP 实验与公网生产环境存在网络路径、延迟和系统负载差异；
 • Windows 本地大量短连接会受到临时端口与 TIME_WAIT 限制，因此实验使用持久连接复用
   请求；
 • Python asyncio 和 Python 线程实验结论聚焦模型边界，不用于预测 Go、Java、C++ 的具
   体性能；
 • RSS 采样会受到 Python 解释器、内存分配器和系统后台活动影响；
 • CPU-bound 实验受 Python GIL 影响，主要用于说明 “协程不创造 CPU 算力”，不用于评
   价其他语言的多线程 CPU 性能。
这些局限明确了结论的适用范围：协程适合降低 I/O 等待任务的线程占用，CPU 密集任务需要并行
计算方案。
