## 0. 一句话结论

协程不是让 CPU 计算自动变快的技术，而是让“等待 I/O 的并发任务”变得更便宜。它通过语言或运行时把函数切分成可暂停、可恢复的执行单元，在等待网络、磁盘、数据库、定时器等事件时释放执行线程，从而用更少的 OS 线程承载更多并发请求。

对 15 分钟报告，建议反复强调三点：

- 协程解决的是高并发 I/O 下的资源浪费，不是替代所有线程和并行计算。
- 协程的底层关键是“状态保存 + 调度恢复 + I/O事件通知”，操作系统仍然只调度真实线程。
- 不同语言的“协程”抽象层级不同：C++20偏底层机制，Python/C#偏 async/await，Go/Java 虚拟线程更接近轻量线程化的工程体验。

## 1. 为什么并发模型演进到协程

### 1.1 背景：云服务的主要矛盾是大量等待

现代服务端应用经常不是被单个请求的计算量压垮，而是被大量请求同时处于等待状态压垮。典型等待包括：

- 等待客户端网络包到达。
- 等待数据库、Redis、消息队列、RPC 下游响应。
- 等待磁盘或对象存储 I/O。
- 等待定时器、限流器、连接池资源。

如果每个请求都绑定一个 OS 线程，线程在等待 I/O 时通常处于阻塞状态。阻塞本身并不消耗很多 CPU，但会占用线程栈、线程对象、调度器元数据，并给内核调度带来更高成本。当并发数从几百上升到几千、几万时，这些成本会变成瓶颈。

Dan Kegel 的 C10K 问题较早集中讨论了“单机同时处理一万个客户端连接”的挑战，核心矛盾就是高并发连接下传统阻塞式线程模型和 I/O 模型的扩展性问题。这个问题后来推动了事件驱动服务器、epoll/kqueue/IOCP、异步框架和协程运行时的发展。

### 1.2 并发执行模型演进

| 阶段 | 核心模型 | 优点 | 主要问题 |
|---|---|---|---|
| 多进程 | 一个请求或一组请求由进程处理 | 隔离强，崩溃影响小 | 创建、切换、内存成本高 |
| 多线程 | 一个请求绑定一个线程 | 代码直观，共享内存方便 | 大量线程带来栈内存、上下文切换和调度开销 |
| 事件循环 | 少量线程处理大量事件 | 线程少，I/O吞吐高 | 回调链复杂，状态分散，可读性差 |
| async/await 协程 | 用顺序代码表达异步等待 | 兼顾性能和可读性 | 需要非阻塞生态，阻塞调用会破坏调度 |
| 虚拟线程/轻量线程 | 用近似线程的API表达高并发任务 | 迁移成本低，工程心智简单 | 运行时仍有调度、pinning、阻塞兼容等边界 |

这一演进不是“旧技术完全淘汰”，而是根据问题类型选择合适抽象：

- CPU 密集：需要真正并行，重点是多核、线程池、进程池、SIMD、GPU。
- I/O 密集：重点是减少等待时占用的 OS 线程，协程和异步 I/O 更合适。
- 强隔离任务：进程仍然有价值。
- 传统同步代码迁移：Java 虚拟线程这类轻量线程抽象很有吸引力。

### 1.3 PPT 可用核心表达

建议在 Slides 中把这一页做成“等待成本”的图：

- 左侧：线程模型。请求 A/B/C/D 每个占一个线程，等待数据库时线程被挂起。
- 右侧：协程模型。请求 A/B/C/D 是多个协程，等待时挂起协程，线程继续执行其他可运行协程。
- 页脚结论：协程把“等待”从占用线程变成占用一个轻量任务对象。

## 2. 协程技术介绍与底层机制

### 2.1 协程的基本定义

协程可以通俗理解为“可暂停、可恢复的函数”。普通函数调用后会沿着调用栈一直运行到返回；协程函数则可以在某些挂起点保存当前执行状态，然后把控制权交回调度器，之后再从挂起点继续执行。

一个协程系统通常包含以下概念：

- 协程函数：用 `async def`、`async`、`co_await`、`go` 等语法或库 API 创建。
- 协程对象/任务对象：代表一次尚未完成的异步执行。
- 挂起点：例如 `await`、`yield`、channel 阻塞、sleep、I/O等待。
- 调度器：决定哪个可运行协程获得线程继续执行。
- 事件源：网络 I/O、定时器、文件 I/O、锁、channel、Future完成等。

### 2.2 Stackless 与 Stackful

协程实现大体可以分成两类：

| 类型 | 特点 | 代表 |
|---|---|---|
| Stackless coroutine | 不保存完整调用栈，编译器把函数拆成状态机，局部变量保存在协程帧中 | C++20 coroutine、Python async coroutine、C# async/await |
| Stackful coroutine / fiber | 每个协程有自己的栈，可在更深层调用中挂起 | 一些 fiber 库、部分语言运行时、虚拟线程内部实现思路 |

Stackless 的优点是轻量、可由编译器明确转换；缺点是函数调用链上需要显式传播 `async/await`。Stackful 的优点是更接近线程心智，调用链侵入性低；缺点是栈管理、调度和运行时复杂度更高。

### 2.3 编译器如何把协程变成状态机

以 async/await 类语言为例，编译器通常会做这些转换：

1. 找到协程函数中的挂起点。
2. 把函数局部变量、下一步执行位置、返回值、异常等保存到协程帧或状态机对象。
3. 遇到 `await` 时，如果等待对象尚未完成，则注册回调或恢复句柄，然后返回给调度器。
4. 等待对象完成后，调度器重新调用恢复函数，从保存的位置继续执行。

因此，从底层看，协程并不神秘，它是“语法糖 + 状态机 + 运行时调度”的组合。

对 PPT，可以画成：

```text
async 函数源码
  -> 编译器识别 await 点
  -> 生成状态机/协程帧
  -> 运行时把任务放入队列
  -> I/O完成后恢复执行
```

### 2.4 运行时调度器做什么

协程运行时一般负责：

- 维护就绪队列：保存可以继续运行的协程。
- 对接事件循环：监听定时器、网络 I/O、Future 完成。
- 分配执行线程：把可运行协程映射到一个或多个 OS 线程。
- 处理取消与超时：传播取消信号，清理资源。
- 处理异常：让异常能回到调用者或任务收集点。
- 提供结构化并发：例如任务组、scope、context，避免任务泄漏。

协程调度通常是协作式的：协程只有在明确挂起点才让出执行权。因此，协程代码中如果出现长时间 CPU 循环或阻塞式调用，就会阻塞事件循环或调度线程。

### 2.5 操作系统仍然只调度线程

一个常见误解是“协程绕过了操作系统”。更准确的说法是：

- 操作系统调度的是内核线程或用户进程，不直接理解 Python Task、C++ coroutine、Go goroutine、Java virtual thread 这些高层任务。
- 协程运行时在用户态保存和恢复协程状态。
- 当协程等待 I/O 时，运行时把 I/O 注册给 OS 的事件通知机制。
- I/O 完成后，OS 通知运行时；运行时再把对应协程放回就绪队列。

不同平台的事件通知机制包括：

- Linux：`epoll`，适合监控大量文件描述符事件。
- BSD/macOS：`kqueue`。
- Windows：I/O Completion Ports，常用于高性能异步 I/O。
- 跨平台库：libuv 封装不同平台的事件循环和异步 I/O能力，Node.js 和 uvloop 等生态与它有关。

### 2.6 协程的优势与代价

优势：

- 任务对象比 OS 线程轻量，能承载更多并发连接。
- 等待 I/O 时不独占线程，线程利用率更高。
- async/await 让异步代码保持接近同步的顺序结构。
- 对服务端高并发 I/O、爬虫、网关、RPC聚合等场景特别有效。

代价：

- 调用链容易“染色”：一个函数变 async，上游往往也要 async。
- 调试栈可能不连续，异常传播和取消传播更复杂。
- 阻塞调用会破坏事件循环，例如在 asyncio 中直接调用 `time.sleep()`。
- 无限制创建协程会造成任务堆积、内存上涨、下游过载。
- CPU 密集任务不会因为协程而突破核心数限制。

## 3. 主流编程语言的协程支持与对比

### 3.1 C++20 coroutine：底层机制而非完整框架

C++20 引入 `co_await`、`co_yield`、`co_return` 等协程语法。它的特点是“给机制，不直接给高级运行时”：

- 编译器负责把 coroutine 转换为可暂停/恢复的协程帧。
- `promise_type`、awaiter、handle 等机制允许库作者高度定制调度、返回类型、内存分配。
- 标准语言层面没有像 Python asyncio 那样统一的事件循环。
- 工程使用通常依赖第三方库或平台库，例如 cppcoro、asio、folly、Qt coroutine 支持等。

适合场景：

- 高性能网络库、游戏服务器、底层异步框架。
- 需要精确控制内存、调度、生命周期的系统编程。

风险：

- 学习曲线陡峭。
- 生命周期、异常、取消、协程句柄管理容易出错。
- “能写出高性能”不等于“默认就容易写对”。

### 3.2 Go goroutine：语言级并发和运行时调度

Go 的 `go f()` 会启动一个 goroutine。Go 官方文档刻意使用 goroutine 这个词，因为它既不是传统线程，也不只是简单协程。goroutine 由 Go runtime 调度，通常配合 channel、context、select 使用。

关键点：

- goroutine 初始栈小，并可按需增长。
- Go runtime 使用 M/P/G 模型组织调度：G 代表 goroutine，M 代表 OS 线程，P 代表处理器资源上下文。
- channel 用于通信和同步，鼓励“通过通信共享内存”。
- `context.Context` 常用于请求级取消、超时和跨 API 传播。

适合场景：

- 高并发网络服务。
- RPC、网关、消息处理、流水线任务。
- 需要简单并发语法和成熟运行时的工程项目。

注意点：

- goroutine 很轻量，但不是免费。无限创建 goroutine 会导致内存和调度压力。
- goroutine 泄漏是 Go 服务端常见问题，尤其是 channel 阻塞、context 未取消、下游永不返回。
- CPU 密集 goroutine 仍受核心数和 GOMAXPROCS 约束。

### 3.3 Python asyncio：显式事件循环和协作式任务

Python 的 `asyncio` 通过 event loop、coroutine、Task、Future 组织异步并发。`async def` 定义协程函数，调用后得到协程对象，通常需要用 `await`、`asyncio.create_task()` 或 `TaskGroup` 执行。

关键点：

- `await` 是明确挂起点，协程在这里把控制权还给事件循环。
- Task 用来把 coroutine 排入事件循环并发执行。
- Python 3.11 引入 `TaskGroup`，提供更接近结构化并发的任务管理方式。
- 适合 I/O 密集任务，例如 HTTP 客户端、WebSocket、爬虫、异步 Web 框架。

注意点：

- asyncio 是协作式调度，CPU 长循环会阻塞事件循环。
- 阻塞库必须放进线程池/进程池，或替换为异步库。
- Python GIL 使多线程 CPU 并行受限，CPU 密集任务通常考虑 multiprocessing、native extension 或外部计算系统。

### 3.4 C# async/await：Task-based 异步模型

C# 的 async/await 与 .NET 的 Task 模型结合紧密。编译器会把 async 方法转换成状态机，await 未完成任务时让出执行权，任务完成后继续执行。

关键点：

- `Task`、`Task<T>` 是异步操作的核心返回类型。
- async 方法看起来像同步流程，但底层是状态机和 continuation。
- .NET 生态中数据库、HTTP、文件、UI 等大量 API 都提供 async 版本。
- 在服务端，async/await 有助于减少阻塞线程，提高吞吐。

注意点：

- async 不是自动开新线程，I/O 异步通常依赖 OS 和运行时通知。
- 不应在异步代码中随意 `.Result` 或 `.Wait()`，可能造成线程阻塞甚至死锁。
- CPU 密集任务仍应显式使用线程池或并行计算库。

### 3.5 Java 21 Virtual Threads：轻量线程化路线

Java 21 通过 JEP 444 正式引入虚拟线程。它的定位不是传统 async/await，而是让“一个任务一个线程”的同步编程风格在高并发 I/O 场景下重新具备扩展性。

关键点：

- 虚拟线程是 JDK 提供的轻量线程，由 JVM 调度到平台线程执行。
- 当虚拟线程执行可卸载的阻塞 I/O 时，JVM 可以把它从承载线程上卸载，让平台线程去执行其他虚拟线程。
- 对很多传统阻塞式 Java 代码，迁移到虚拟线程比改写成 CompletableFuture 链更容易。
- 目标是保留线程式代码结构，同时降低每任务绑定平台线程的成本。

注意点：

- 虚拟线程不是为了加速 CPU 密集任务。
- 某些场景下虚拟线程可能被 pin 到平台线程，例如涉及特定同步或 native 调用时，需要关注 JDK 版本和诊断工具。
- 数据库连接池、下游服务容量、限流仍然必须控制。虚拟线程让并发更便宜，但不会让下游无限扩容。

### 3.6 横向对比表

| 语言/平台 | 抽象形态 | 调度主体 | 代码风格 | 适合场景 | 主要风险 |
|---|---|---|---|---|---|
| C++20 | 底层 coroutine 机制 | 库/框架决定 | `co_await` 等 | 高性能系统库 | 复杂、生命周期难 |
| Go | goroutine + channel | Go runtime | `go f()` | 服务端高并发 | goroutine 泄漏、无界并发 |
| Python | asyncio Task | event loop | `async/await` | I/O密集脚本和服务 | 阻塞事件循环、GIL |
| C# | Task async/await | .NET runtime | `async/await` | 应用层异步 I/O | `.Result`/`.Wait()` 阻塞 |
| Java 21 | virtual thread | JVM | 同步线程式代码 | 传统服务端迁移 | pinning、下游容量控制 |

### 3.7 选型结论

如果是课程报告，不建议把所有语言讲成“都差不多”。更好的表达是：

- C++20：给框架作者和系统程序员的底层能力。
- Go：把轻量并发作为语言默认工具。
- Python：把事件循环和 async/await 显式暴露给应用层。
- C#：围绕 Task 生态构建的成熟应用异步模型。
- Java：用虚拟线程降低传统同步服务端代码迁移成本。

### 4.3 I/O 等待实验结果

在 10000 个等待任务、每个等待 0.01 秒的场景中：

| 模型 | worker数 | 平均耗时(s) | 吞吐(tasks/s) | Python分配峰值(MB) | Python线程峰值 |
|---|---:|---:|---:|---:|---:|
| asyncio | 无 | 1.864 | 5364.71 | 15.11 | 1 |
| threads | 50 | 2.935 | 3406.90 | 17.50 | 51 |
| threads | 100 | 2.202 | 4541.53 | 17.24 | 101 |
| threads | 200 | 2.157 | 4636.05 | 16.69 | 201 |

可得结论：

- asyncio 只用 1 个主要执行线程，就完成了 10000 个等待任务。
- 多线程需要 50/100/200 个 worker 才能逼近 asyncio 的耗时。
- 随着线程数上升，耗时下降，但线程资源成本明显增加。
- 协程优势来自“等待时不占线程”，不是让 sleep 本身变快。

- 用 `assets/coroutine_exp_io_threads.png` 展示线程数量差异，这张图最能让听众直观看到协程价值。
- 讲解时强调：这是 I/O 模拟，不代表 CPU 计算。

### 4.4 CPU-bound 实验结果

在 200 个 CPU 计算任务中：

| 模型 | worker数 | 平均耗时(s) | 吞吐(tasks/s) | Python分配峰值(MB) | Python线程峰值 |
|---|---:|---:|---:|---:|---:|
| asyncio | 无 | 6.811 | 29.36 | 0.159 | 1 |
| threads | 50 | 6.696 | 29.87 | 0.396 | 27.4 |
| threads | 100 | 6.995 | 28.59 | 0.406 | 30.2 |
| threads | 200 | 7.093 | 28.20 | 0.398 | 28.0 |

可得结论：

- CPU-bound 场景下，asyncio 没有明显优势。
- Python 多线程也没有显著加速，原因与 GIL 和线程调度开销有关。
- 线程数增加并不一定带来更快结果，反而可能增加调度开销。
- CPU 密集任务应该考虑进程池、C/C++扩展、NumPy向量化、GPU或其他并行计算方案。

PPT建议：

- 用 `assets/coroutine_exp_cpu_duration.png` 展示 CPU 场景下各模型耗时接近。
- 这一页用于纠偏：协程适合管理等待，不适合创造算力。

### 4.5 实验结论边界

实验可以支撑的结论：

- 在大量短等待任务中，asyncio 能用更少线程承载更高并发。
- 线程池要提高 I/O 等待吞吐，通常需要增加 worker 数。
- 在 CPU-bound 场景中，协程不会自动带来并行加速。

实验不能直接支撑的结论：

- 不能断言所有语言协程都一定快于线程。
- 不能断言真实网络服务中的延迟一定与 sleep 模拟一致。
- 不能用 `tracemalloc` 结果替代系统级 RSS 内存结论。
- 不能用 Python 线程实验直接代表 Go、Java、C++ 的运行时表现。

如果还有时间，可追加更强实验：

- 使用真实 HTTP server/client，测 QPS 和 P95 延迟。
- 使用 `psutil.Process().memory_info().rss` 记录进程 RSS。
- 使用 Linux 环境记录 context switches。
- 对比 `asyncio`、`aiohttp`、`requests + ThreadPoolExecutor`。

## 5. 工程应用范式与常见坑

### 5.1 适合协程的场景

- 高并发 Web 服务。
- API 网关和 RPC 聚合。
- 爬虫与批量 HTTP 请求。
- 数据库、缓存、消息队列等 I/O 密集访问。
- 长连接服务，例如 WebSocket、IM、推送。
- 大量定时器或超时控制任务。

### 5.2 不适合把协程当核心优化的场景

- 纯 CPU 计算。
- 依赖大量阻塞式第三方库，且无法替换或隔离。
- 强实时调度任务。
- 需要进程级隔离、安全边界或崩溃隔离的任务。

### 5.3 协程工程实践清单

应当做：

- 给每个外部调用设置超时。
- 用结构化并发管理任务生命周期，例如 Python `TaskGroup`、Go `context`、Java structured concurrency 相关 API。
- 对并发数设置上限，例如 semaphore、连接池、令牌桶。
- 明确取消传播路径。
- 区分 I/O-bound 和 CPU-bound，CPU任务移出事件循环。
- 建立指标：任务数、队列长度、事件循环延迟、超时率、取消率、下游错误率。

避免：

- 无限 `create_task()` 或无限启动 goroutine。
- 在事件循环中直接调用阻塞函数。
- 忽略取消后资源清理。
- 只测平均耗时，不看 P95/P99 延迟。
- 误把协程当成“比线程更快的计算单元”。

## 7. 可直接放入 Slides 的关键句

- 协程的目标不是让单个任务更快，而是让大量等待中的任务更便宜。
- 操作系统调度线程，运行时调度协程；协程没有脱离线程存在。
- `await` 的本质是：保存状态，把线程还给调度器，等事件完成后恢复。
- 线程模型的直观性和事件循环的高吞吐，在协程中被重新组合。
- CPU密集任务需要并行计算，I/O密集任务需要便宜的等待。
- 虚拟线程说明了一个趋势：语言和运行时正在把高性能异步能力重新包装成更容易理解的同步风格。
