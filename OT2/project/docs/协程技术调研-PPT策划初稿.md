# 协程技术调研 PPT 策划初稿

> 定位：这是进入视觉设计和正式制作为 PPTX 之前的策划稿。目标不是堆资料，而是明确 15 分钟报告应该让听众记住什么、每页证明什么、用什么图表或结构承载信息。

## 1. 汇报目标

### 1.1 听众应该带走的核心认知

这份报告希望听众记住：

- 协程不是“更快的线程”，而是让大量 I/O 等待任务更便宜的执行抽象。
- 协程的底层关键是：保存状态、让出线程、等待事件、恢复执行。
- 操作系统仍然只调度线程；协程由语言、编译器和运行时在用户态组织。
- 不同语言的协程支持差异很大，不能只看语法名词。
- 实验结论要有边界：I/O 等待中协程有优势，CPU 密集中协程不创造算力。

### 1.2 总体叙事线

```text
高并发服务的瓶颈不是一直在算，而是大量任务在等
  -> 线程模型让等待变贵
  -> 事件循环让等待便宜但代码变复杂
  -> 协程把事件驱动性能重新包装成顺序代码体验
  -> 语言和运行时以不同方式实现这种抽象
  -> 实验验证：协程适合 I/O 等待，不适合替代 CPU 并行
```

### 1.3 报告基调

- 技术调研型，不做营销式介绍。
- 每页一个结论，图表或机制图支撑结论。
- 避免长段文字，优先使用流程图、对比表、实验图。
- 语言上保持“先结论、后解释、再边界”。

## 3. 视觉设计方向

### 3.1 推荐风格

主题是系统并发与运行时机制，建议采用“工程白板 + 深色技术底”的混合风格：

- 背景：浅灰白或深墨色交替，机制页可用深色，图表页用浅色。
- 视觉重点：线框流程、队列、线程轨道、状态机节点、实验图。
- 避免：大量圆角卡片、渐变背景、装饰性图标堆叠。
- 页面密度：中等偏高，但每页只保留一个主证明对象。

### 3.3 图表与图形规则

- 图表标题必须是结论，不写“实验结果一”这类空标题。
- 图表优先直接标注关键数值，减少图例依赖。
- 机制图使用一致语法：
  - 蓝色：协程/任务。
  - 红色：OS线程。
  - 绿色：运行时调度器/事件循环。
  - 灰色：操作系统与I/O事件。
- 表格最多保留 5 行 6 列以内，超出就拆页或转为矩阵。
- 每页底部保留很轻的来源脚注，避免影响阅读。

### Slide 03. 演进页：并发模型一步步把等待成本降下来

**页面结论**

并发模型演进表。

**画面设计**

- 每一阶只放一行收益和一行代价。

**讲述重点**

### Slide 04. 关键转折：事件循环性能好，但协程把代码可读性带回来

**页面结论**

**画面设计**

**讲述重点**

让听众理解协程的出现不是单纯追求性能，也是在修复异步代码的可维护性问题。

### Slide 05. 定义页：协程是可暂停、可恢复的函数

**页面结论**

协程的核心不是“并行”，而是执行过程可以在挂起点被切开。

**证明对象**

普通函数 vs 协程函数执行轨迹图。

**内容结构**

- 普通函数：调用 -> 连续执行 -> 返回。
- 协程函数：调用 -> 执行一段 -> await 挂起 -> 恢复 -> 返回。
- 协程对象保存：局部变量、下一步位置、返回值、异常、恢复句柄。

**画面设计**

- 上下两条轨道。
- 普通函数轨道是一条连续实线。
- 协程轨道是分段线，中间有 `await I/O` 的暂停节点。

**讲述重点**

这一页要把“可暂停、可恢复”讲透，后面状态机和调度器才容易理解。

**预计时长**

1.0 min

### Slide 06. 底层机制：协程本质是状态机 + 调度恢复

**页面结论**

编译器或解释器把协程函数拆成状态机，运行时负责在事件完成后恢复它。

**证明对象**

状态机转换图。

**内容结构**

```text
async源码
  -> 识别 await 点
  -> 生成协程帧/状态机
  -> await 未完成时注册恢复逻辑
  -> 事件完成后从保存状态继续执行
```

**画面设计**

- 左侧放伪代码：
  - `async def fetch():`
  - `data = await read()`
  - `return parse(data)`
- 右侧放状态机：
  - State 0: start
  - State 1: waiting read
  - State 2: parse and return
- 中间用箭头连接“编译器/解释器转换”。

**讲述重点**

不要陷入某个语言细节，讲通用机制：保存状态，稍后恢复。

**预计时长**

1.2 min

### Slide 13. 工程判断：什么时候该用协程，什么时候不该用

**页面结论**

选择协程前先判断瓶颈：等待多就用协程，计算多就用并行计算。

**证明对象**

决策矩阵。

**内容结构**

| 场景 | 判断 | 推荐 |
|---|---|---|
| 高并发网络请求 | I/O等待占主导 | 协程/异步I/O/虚拟线程 |
| 数据库/RPC聚合 | 大量等待下游响应 | 协程 + 超时 + 限流 |
| CPU计算 | 计算占主导 | 线程池/进程池/SIMD/GPU |
| 阻塞第三方库 | 会卡住事件循环 | 隔离到线程池或换异步库 |
| 无界任务创建 | 容易拖垮下游 | 加 semaphore、连接池、背压 |

**画面设计**

- 中央为决策树：
  - 是否 I/O 等待占主导？
  - 是否能使用非阻塞库？
  - 是否需要限制下游资源？
- 右侧列“必须配套”：超时、取消、限流、监控。

**讲述重点**

协程是工程工具，不是性能魔法。真正可靠的并发系统还需要资源上限和取消机制。

**预计时长**

1.2 min

### Slide 14. 总结与AI使用说明

**页面结论**

理解协程，就是理解语言、运行时和操作系统如何共同降低等待成本。

**总结三句话**

- 协程的价值：让大量 I/O 等待任务不再大量占用 OS 线程。
- 协程的机制：编译器/解释器保存状态，运行时调度恢复，OS通知I/O事件。
- 协程的边界：CPU密集任务需要并行计算，不应放进事件循环硬跑。

- AI用于资料搜集、结构梳理、实验脚本审阅与图表生成辅助。

## 6. 设计阶段注意事项

### 6.1 文案压缩原则

每页只保留：

- 一个结论标题。
- 一个主证明对象。

### 6.4 可能被问到的问题

**问题：协程是不是比线程更快？**

回答：不是绝对更快。协程的优势主要在 I/O等待场景，因为等待时不占用线程。CPU密集任务仍然受核心数、GIL、运行时和调度开销影响。

**问题：Java虚拟线程和协程有什么区别？**

回答：虚拟线程更接近轻量线程化抽象，目标是保留同步线程式代码；async/await 协程通常要求调用链显式异步。二者都在降低等待成本，但工程心智不同。

**问题：实验为什么用 Python？**

回答：Python 的 asyncio 和 ThreadPoolExecutor 易于构造可复现实验，能清楚展示 I/O等待和CPU计算两类边界。但不能把 Python 结果直接推广到 Go、Java、C++ 的具体性能。

**问题：`tracemalloc` 是系统内存吗？**

回答：不是。它统计 Python 层分配峰值，不等于进程 RSS。PPT中应标注为 Python allocation peak。
